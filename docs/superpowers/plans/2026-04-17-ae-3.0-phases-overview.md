## Strategic context ‚Äî the four crystallizations

Every phase serves these. Re-read before starting any phase.

1. **Knowledge is canonical, code is realization.** Canonical = the score; artifact = the performance. Multiple impls (Dart, Rust, Kotlin/Swift) are different performances of one score.
2. **AE composes with the agent rather than competing with it.** Heuristic by default; LLM work is delegated to the host agent (Claude Code Task tool, Codex `exec`, Cursor agent mode). BYOK direct LLM is the headless/CI fallback. AE never owns a model.
3. **Knowledge has two layers: canonical (intent) and artifact (instance).** They reference each other; they're never the same thing.
4. **Canonical is a token-efficient cognitive map, not exhaustive documentation.** ~10‚Äì50 features per pack, ~2‚Äì4k tokens, decompose into siblings when bigger.

Persona: **solo dev maintaining one or more large codebases of their own** (e.g. ecsly: 250K LOC Dart/Rust/Kotlin/Swift, 46 plugins, 8 prototypes). Anti-persona for 3.0: large teams, library publishers selling registry entries.

Scope boundary (spec ¬ß14): NO cross-language code generation; NO auto-sync; NO public canonical hub; NO multi-repo workspaces; NO team flows; NO dgraph backend; NO auto package-hub discovery; NO migration command from old `know/` layouts. Designed-for-but-not-built: package hub layer in resolver, `canonical_remotes` field in hub.yaml.

## Phase 2 ‚Äî Heuristic extractors (Dart deep, Rust solid, Kotlin/Swift best-effort)

> **SHIPPED** ‚Äî tag `ae-3.0-phase-2-complete`. Detailed plan archived at [`archive/2026-04-17-ae-3.0-phase-2-heuristic-extractors.md`](archive/2026-04-17-ae-3.0-phase-2-heuristic-extractors.md). No material deviation from the outline.

**Goal:** Three `HeuristicExtractor` adapters that turn a local source directory into a structural `ArtifactPack` (no LLM). Fast: O(files), seconds even on `dart_flutter_packages`-scale.

### Per-adapter capabilities

- **`DartHeuristicExtractor` (deep):**
  - Detects `pubspec.yaml`; recursive workspace walk if `melos.yaml` present.
  - Parses `pubspec.yaml`: name, description, version, dependencies (split runtime vs dev).
  - Scans `lib/`: barrel file detection (`<package>.dart` + `lib/src/`), library directives, public class/function/typedef enumeration via simple regex (no analyzer dependency in 3.0; analyzer is a fast-follow upgrade).
  - Harvests doc comments (`///`) above public symbols.
  - **Bridge-package detection:** flags packages depending on `ffi` or referencing `MethodChannel` (presence in any source file).
  - README excerpt: first paragraph + first H2 of `README.md` if present.

- **`RustHeuristicExtractor` (solid):**
  - Detects `Cargo.toml`; reads `[package].name`, `description`, `version`.
  - Workspace member walk if `[workspace]` table present.
  - Scans `src/lib.rs` and `src/main.rs`: `pub` items via regex.
  - Feature flags from `[features]`.
  - README excerpt as above.

- **`KotlinSwiftHeuristicExtractor` (best-effort):**
  - Detects `Package.swift` (Swift) or `build.gradle.kts` / `build.gradle` (Kotlin/Android/JVM).
  - Lists Swift/Kotlin source files; counts top-level classes/structs by simple regex.
  - No deep semantic parse in 3.0. Output is a structural inventory + dep list.
  - README excerpt as above.

### Output shape

`HeuristicArtifact` (in-memory) ‚Üí produces:
- `meta.yaml` skeleton (kind=`local`, source path, file SHAs via `crypto`, language, scanned_at, license detected from `LICENSE` file or package manifest).
- `index.md`: package title, README excerpt, exports/public-API summary, dependency list.
- Empty `matrix.yaml` (`features: []`); rows materialized when `ae artifact link --canonical` later (Phase 4).
- No `patterns.md`, no `drift.yaml` yet.

### Integration point in Phase 2

A small `HeuristicExtractorRegistry` class with the dispatch:

```dart
class HeuristicExtractorRegistry {
  HeuristicExtractorRegistry(this._extractors);
  final List<HeuristicExtractor> _extractors;

  Future<HeuristicExtractor?> findFor(final Directory dir) async {
    for (final e in _extractors) {
      if (await e.canHandle(dir)) return e;
    }
    return null;
  }
}
```

The full `ArtifactService.ingest()` orchestration (which uses this registry, walks the repo, saves via `ArtifactStore`) lands in **Phase 4** alongside the CLI command. Phase 2 ships only the extractors and their tests.

### Executor contracts

Each implements:
```dart
abstract interface class DistillationExecutor {
  String get executorId;                            // "claude_code" | "codex" | "byok"
  Future<bool> canRun();                            // detect host availability now
  Future<DistillationOutput> execute(DistillationTask task);
}
```

### Host detection (in `canRun()`)

- **Claude Code subagent:** detect via env vars (`CLAUDE_CODE_VERSION`, `CLAUDECODE`, etc.) or parent-process inspection. When in Claude Code, call the Task tool via the MCP-bridged dispatch (or shell to `claude -p` for non-MCP contexts). Zero config required when host is present.
- **Codex exec:** detect `CODEX_HOME` / `OPENAI_CODEX_VERSION`. Shell to `codex exec` with the task as JSON input on stdin.
- **BYOK LLM:** detect `byok:` block in `hub.yaml` (provider + key/env-var name). Direct API call. Used in CI/headless or when user prefers it.

### Validation + retry

`DistillationService.distillFromArtifact()`:

1. Build `DistillationTask` from the artifact (load it, gather files, compose seed rows, attach examples).
2. Pick executor: first whose `canRun()` returns true, in priority order: `claude_code` ‚Üí `codex` ‚Üí `byok`.
3. Execute. On schema validation failure (output not a valid `DistillationOutput`), retry once with the validation error included as additional context. Second failure ‚Üí fail with diagnostic; no silent partial merge.
4. Merge into canonical via `CanonicalService.upsert(conceptId, output)`.

### Cost preview

`ae canonical distill --estimate`:
- For BYOK executor: returns input token count (cheap server-side or local tokenizer estimate).
- For Claude Code / Codex executors: returns "this dispatches N subagent task(s); cost is your subscription's." No dollar estimate.

### Acceptance

- All three executors implement the port; mocked tests round-trip a task.
- Dispatcher correctly selects based on (mocked) env detection.
- Schema-fail retry behavior tested.
- Tag `ae-3.0-phase-3-complete`.

~12 tasks ¬∑ ~60 TDD steps. Subagent dispatch + mocked HTTP are the trickier parts.

#### 4B ‚Äî Cross-pack verify + project-wide cockpit

`ArtifactService.verifyProject()`:
- Walk all artifacts.
- For each artifact: walk its `requires:` graph.
- Compute downstream-demand counts per (artifact, feature_id).
- Output the **tier-classified gap report**:
  - **Tier 1 ‚Äî invariant violations** (canonical asserts; no test verifies)
  - **Tier 2 ‚Äî upstream blockers** (downstream-required features missing/partial)
  - **Tier 3 ‚Äî partial referenced features**
  - **Tier 4 ‚Äî unreferenced canonicals** (with "new since last verify" badge)
- Filtered by default; `--show all` for firehose.

`ArtifactService.verifyOne(packName, --strict)`:
- Same scheme, scoped to one artifact.
- `--strict` exits non-zero on Tier 1+2 unless explicitly `accepted` in `drift.yaml`.

#### 4C ‚Äî Drift detection

`DriftService.computeCodeDrift(packName)`:
- Compare current file SHAs in source path against `meta.source.files`.
- Output `code_drift` entries in `drift.yaml`.

`DriftService.computeIntentDrift(packName)`:
- For each canonical referenced by the artifact:
  - For each canonical feature with an `invariant` field:
    - Check if the artifact has a matrix row with `tests: yes` for that feature.
    - If not ‚Üí `intent_drift` entry: "invariant unverified."
