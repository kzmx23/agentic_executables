# AE 3.0 — Design

**Status:** Implemented through v3.1.0 (2026-04-27). See [`CHANGELOG.md`](../../../CHANGELOG.md) for the
release-by-release breakdown. This spec is preserved as the design-time record;
for current behavior consult [`docs_site/docs/ae-3/`](../../../docs_site/docs/ae-3/). Sections below carry
inline `**SHIPPED**` / `**DEFERRED**` / `**CHANGED**` markers added 2026-04-27
reflecting state at v3.1.0.

**Date:** 2026-04-17
**Owner:** @arenukvern
**Supersedes:** parts of 2.x architecture (`know/` flat layout; v1 registry framing)

---

## 1. Vision

> **AE 3.0 turns your codebase into a navigable knowledge spine.**
> Heuristic extraction maps any repo into structured artifacts in seconds.
> Canonical packs hold your design intent — language-agnostic, versioned, eventually shareable.
> Drift detection keeps code, artifacts, and intent honest.
> The agent you already use (Claude Code, Codex, Cursor) does any heavy thinking via a clean delegation contract.
> AE never owns a model, never charges per token, never replaces your tools — it makes them sharper.

## 2. Principles (the four crystallizations)

1. **Knowledge is canonical, code is realization.** The same canonical knowledge can describe a Dart impl and a Rust impl as two performances of one score.
2. **AE composes with the agent rather than competing with it.** Heuristic by default; LLM work is delegated to the host agent (Claude Code Task tool, Codex `exec`, Cursor agent mode), with BYOK direct-LLM as the fallback for headless / CI.
3. **Knowledge has two layers: canonical (intent) and artifact (instance).** Canonical = the score. Artifact = the performance. They reference each other; they're never the same thing.
4. **Canonical is a token-efficient cognitive map, not exhaustive documentation.** A pack targets ~10–50 features and ~2–4k tokens. Complex specs decompose into multiple sibling packs that compose. Documentation tools already exist; this is something different.

## 3. Persona

Solo dev maintaining one or more large codebases of their own (e.g. a 250K-LOC Dart/Rust/Kotlin/Swift game engine with 46 plugins and 8 prototypes), using AE as the knowledge spine that keeps them and their AI agents aligned across sessions, for both greenfield ("finish a prototype") and maintenance ("plugin drift across the engine").

Anti-persona for 3.0: large engineering teams with shared registries (post-3.x). Library publishers selling install-instructions to public consumers (still works through `ae registry get`, but not the focus).

## 4. Data Model

### 4.1 Hub layout

> **SHIPPED** in 3.0.0.
> **CHANGED in 3.0.2:** `ae hub init --path X` originally created `know/ packages/ use/ hub.yaml` directly inside `X`; 3.0.2 always nests under `<resolved>/.ae_hub/` and scaffolds the v3 layout (`canonical/`, `artifacts/{local,external,use}/`) per this section. (Iter 0 dogfood bug 1.)

Hubs are the unit of storage. Three hub kinds, resolved in order:

1. **Project hub** `<repo>/.ae_hub/` — always wins for artifacts. Project-private canonical also lives here.
2. **User hub** `~/.ae_hub/` — canonical shared across all your projects.
3. **Package hub** `<pkg>/.ae_hub/` — canonical shipped with packages. **3.0 designs the resolution chain to support it; auto-discovery is roadmapped to 3.x.** Manual: `ae canonical import --from <pkg>/.ae_hub/canonical/<concept>` copies into the project hub.
4. **Remote hub** — public canonical hub. Roadmapped post-3.0.

#### Project hub layout

```
.ae_hub/
  hub.yaml                            # config; reserves canonical_remotes for post-3.0 public hub
  canonical/
    <concept>/                        # live canonical (current major)
      matrix.yaml
      index.md
      meta.yaml                       # version: <int>; license; authors; sources; provenance
      CHANGELOG.md                    # human-readable history of in-place edits
      v1/                             # frozen snapshot — created ONLY at the v2 bump
        matrix.yaml
        index.md
        meta.yaml
        migration_to_v2.md            # optional, encouraged
  artifacts/
    local/<pack_name>/                # extracted from this repo
      meta.yaml                       # source path; file hashes; refs to canonical/; license; authors
      index.md                        # heuristic structural summary (or distilled if upgraded)
      matrix.yaml                     # cells filled in: which canonical features satisfied + status
      patterns.md                     # impl-specific idioms (only present if distilled)
      drift.yaml                      # written by `ae sync`; not by hand
    external/<pack_name>/             # extracted from URLs / PDFs / foreign repos
      ...same shape as local
    use/<library_id>/                 # ae_install/uninstall/update/use.md (yours or others')
      meta.yaml
      ae_install.md
      ae_uninstall.md
      ae_update.md
      ae_use.md
```

#### User hub layout

```
~/.ae_hub/
  hub.yaml
  canonical/
    <concept>/                        # canonical you reuse across projects
```

`packages/` (Lythe instructions) is **not** a 3.0 artifact kind. Slot it in later as another `kind` if a concrete need arises.

### 4.2 Canonical pack

> **SHIPPED** in 3.0.0.
> **CHANGED in 3.0.2:** `mergeDistillation` now widens `column_schema` to include any cell keys observed on merged features (first-seen order, type `text`), so `canonical/<concept>/matrix.yaml` is always self-consistent on both first-write and merge paths. Resolved a scaffold-vs-distill mismatch where `[spec, invariant]` schema co-existed with `invocation`/`notes` cells. (Iter 0 dogfood bug 5.)

A canonical pack describes a load-bearing concept in a language-agnostic way.

```yaml
# canonical/<concept>/meta.yaml
schema: ae.canonical.meta.v1
concept: ecs                          # slug; namespaced for project-private (e.g. ecsly/render_pipeline)
version: 1                            # current major; integer
title: "Entity-Component-System (canonical)"
license:
  spdx: "CC-BY-4.0"                   # required; default suggestion for docs
  url: "https://creativecommons.org/licenses/by/4.0/"
authors:
  - name: "Anton Malofeev"
    role: original_author             # original_author | contributor | maintainer
sources:                               # references that informed this pack; required
  - kind: paper                       # paper | website | code | book | spec
    title: "Data-Oriented Design Book"
    url: "https://www.dataorienteddesign.com/dodbook/"
  - kind: code
    title: "Bevy ECS"
    url: "https://github.com/bevyengine/bevy"
provenance:
  authored: hand                      # hand | distilled_from_artifact | imported_from_public_hub
  authored_at: "2026-04-17"
  distilled_from: null
```

```yaml
# canonical/<concept>/matrix.yaml
schema: ae.canonical_matrix.v1
concept: ecs
version: 1
column_schema:                        # canonical only declares schema; cells live in artifacts
  - { id: spec, type: text }
  - { id: invariant, type: text }
  - { id: test_recipe, type: text }   # how to verify the invariant; optional
features:
  - id: entity.create                 # stable id; namespaced; never reused
    spec: "An entity is created with a unique, opaque handle."
    invariant: "Handles are non-reusable within a session."
  - id: system.tick
    spec: "Systems run in declared order each tick."
    invariant: "Tick is monotonically increasing."
```

```markdown
# canonical/<concept>/index.md

A condensed prose model of the concept. ~600 words target.
Object model. Lifecycle. Hard rules. Decomposition pointers to sibling canonicals.
NOT a full reference. NOT exhaustive documentation. The features in matrix.yaml are the contract.
```

**Granularity rule:** 10–50 features per canonical pack; ~2–4k tokens total. Beyond that, decompose into sibling packs (e.g., `gltf/core` + `gltf/extensions/khr_lights_punctual` + `gltf/extensions/khr_materials_clearcoat` + …).

### 4.3 Artifact pack

> **SHIPPED** in 3.0.0.

An artifact pack describes one concrete instance: source, status, drift.

```yaml
# artifacts/<kind>/<pack_name>/meta.yaml
schema: ae.artifact.meta.v1
kind: local                           # local | external | use
title: "Dart ecs (core engine)"
source:
  type: path                          # path | url | git
  path: "core_packages/ecs"           # repo-relative for local
  files:
    - { path: "lib/src/world.dart", sha256: "..." }
    - { path: "lib/src/entities.dart", sha256: "..." }
  scanned_at: "2026-04-17T13:00:00Z"
license:
  spdx: "MIT"
  detected_from: license_file         # license_file | page_metadata | manifest | manual
authors:
  - name: "Anton Malofeev"
    detected_from: pubspec
references_canonical:                 # what this artifact CLAIMS TO IMPLEMENT
  - ecs                               # live (current); resolves to canonical/ecs/
  - ecsly/schedule                    # project-namespaced
  - gltf/core@v2                      # locked to snapshot v2
extractor: dart_v1                    # which HeuristicExtractor produced this
distill: { engine: heuristic, applied_at: null }  # heuristic | inference
```

```yaml
# artifacts/<kind>/<pack_name>/matrix.yaml
schema: ae.artifact_matrix.v1
column_schema:
  - { id: impl, type: enum, values: [done, partial, missing, planned, n_a, deviates] }
  - { id: algorithm, type: text }                  # explicit algo choice (optional)
  - { id: location, type: path }
  - { id: tests, type: enum, values: [yes, no, partial] }
  - { id: notes, type: text }
features:
  - id: entity.create
    canonical: ecs
    impl: done
    location: lib/src/entities.dart:42
    tests: yes
  - id: system.tick
    canonical: ecs
    impl: partial
    notes: "Order respected; monotonicity not asserted in tests"

requires:                             # cross-artifact dependencies (NEW for 3.0)
  - artifact: dart_ecs_render3d_core
    canonical: ecsly/render_pipeline
    features: [render.scene_extract, render.draw_pass, render.instanced]
  - artifact: rust_ecs_physics_rapier
    canonical: ecsly/physics_step
    features: [physics.step, physics.spatial_index]
```

```yaml
# artifacts/<kind>/<pack_name>/drift.yaml
schema: ae.artifact.drift.v1
generated_by: ae sync
generated_at: "2026-04-17T14:00:00Z"
code_drift:                           # files changed since last extract
  - { path: "lib/src/world.dart", change: modified, hash_was: "...", hash_is: "..." }
  - { path: "lib/src/new_module.dart", change: added }
intent_drift:                         # canonical features whose invariants are unverified
  - feature_id: system.tick
    canonical: ecs
    invariant: "Tick is monotonically increasing"
    reason: "no test asserts this"
accepted:                             # hand-marked drift the user has accepted
  - feature_id: mesh.primitive
    note: "Tangents computed per-pixel deliberately; accepted algorithm choice"
```

### 4.4 References — `references_canonical` and `requires` are different things

> **SHIPPED** in 3.0.0.

- **`references_canonical`** = "I CLAIM TO IMPLEMENT these canonical features." Drives matrix row generation and drift-from-intent checks.
- **`requires`** = "I CONSUME features from these named upstream artifacts." Drives cross-artifact verify and project-wide priority.

The same canonical can appear in both for different reasons (a render plugin implements `render_pipeline/v1`; a prototype requires features from it).

**Live vs locked references resolve at `ae sync` time**, not on every command. `references_canonical: ecs` (live) re-resolves to whatever `canonical/ecs/` currently holds when `ae sync` runs, and the artifact's matrix is re-materialized accordingly (with old impl cells preserved by feature id). `references_canonical: gltf/core@v2` (locked) always resolves to `canonical/gltf/core/v2/`. `ae sync` is the deterministic moment of refresh; reads in between are stable.

### 4.5 Versioning — pragmatic, not religious

> **SHIPPED** in 3.0.0.

- Default: canonical packs are **living documents** at `canonical/<concept>/`. No version directory.
- Edits in place are fine for: new features, clarifying spec/invariant prose, adding columns, adding examples.
- **Snapshots** (`v1/` subdirectory) are created **only when breaking**: removing a feature, renaming a feature id, strengthening an invariant beyond conformant impls, changing column schema.
- Snapshot freezes the pre-break state for consumers locked to it. The live pack becomes the new major.
- Artifact references can be **live** (`ecs`) or **locked** (`gltf/core@v2`).
- Future migration tooling (post-3.0) operates on these snapshots.

### 4.6 Attribution — first-class

> **SHIPPED** in 3.0.0.

- **Canonical packs**: `license` required (default suggestion CC-BY-4.0). `sources` required. `authors` encouraged. Public-hub publish (post-3.0) blocks on missing license.
- **External artifacts**: license best-effort (LICENSE file / page metadata); `unknown` if not detectable, with notes flagging the user.
- **Local artifacts**: license detected from package's LICENSE / manifest; `unknown` if absent.
- **Sources** required for canonical and external artifacts; for local artifacts the source is the repo path.

## 5. Architecture — Ports and Adapters

> **SHIPPED** in 3.0.0–3.0.1 (Phase 1 foundation + Phase 4F hard-cut deletes of `know.dart`/`file_know_store.dart`).

Three independent adapter families. Each axis is independently extensible.

### 5.1 Family: `KnowledgeSource` (existing, extended)

> **SHIPPED** in 3.0.0; renames (`passthrough_extractor.dart` → `passthrough_source.dart`, `url_extractor.dart` → `url_html_source.dart`) landed in the 3.0.1 hard cut.

How to get raw content from somewhere.

```dart
abstract interface class KnowledgeSource {
  bool canHandle(KnowSourceSpec spec);
  Future<RawContent> fetch(KnowSourceSpec spec);
}
```

3.0 adapters: `passthrough_source`, `url_html_source`, `pdf_source`, `git_clone_source` (all carry over from 2.x with minor extensions).

### 5.2 Family: `HeuristicExtractor` (NEW for 3.0)

> **SHIPPED** in 3.0.0 (Phase 2). Dart deep, Rust solid, Kotlin/Swift best-effort all implemented.

How to parse a local source directory into a structural artifact, with no LLM.

```dart
abstract interface class HeuristicExtractor {
  String get languageId;                    // "dart" | "rust" | "kotlin_swift"
  bool canHandle(Directory sourceDir);
  Future<HeuristicArtifact> extract(Directory sourceDir);
}
```

3.0 adapters:
- **`DartHeuristicExtractor`** (deep): pubspec.yaml workspace detection, recursive sub-package walk, barrel-file detection, library directive parsing, public symbol extraction, doc-comment harvesting, bridge-package detection (presence of `ffi` / methodChannel).
- **`RustHeuristicExtractor`** (solid): Cargo.toml workspace members, `pub` item enumeration, feature flag awareness.
- **`KotlinSwiftHeuristicExtractor`** (best-effort day-one): parses `Package.swift` / `build.gradle.kts`; lists Kotlin/Swift class files; no deep semantic parse.

JS/TS adapter is fast-follow when a project needs it; not in 3.0.

A `HeuristicArtifact` produces:
- `meta.yaml` skeleton (source, file hashes, language, scanned_at)
- `index.md`: package title, README excerpt (first paragraph), parsed exports/public API summary, dependency list from manifest
- Empty `matrix.yaml` (`features: []`); rows are added when canonical references are linked

### 5.3 Family: `DistillationExecutor` (NEW for 3.0)

> **SHIPPED** in 3.0.0 (Phase 3). All three adapters (`claude_code`, `codex`, `byok`) ship.
> **CHANGED in 3.1.0:** `DistillationExecutor.execute` still returns `DistillationOutput`, but `DistillationService.distill` now returns `DistillationResult { output, executorId }` (see §7) so CLI/MCP envelopes report `executor_used` directly from the result rather than re-running `canRun()` on every executor.

How to hand a distillation task to an agent and receive validated JSON.

```dart
abstract interface class DistillationExecutor {
  String get executorId;                    // "claude_code" | "codex" | "byok"
  bool canRun();                            // is host available now?
  Future<DistillationOutput> execute(DistillationTask task);
}
```

3.0 adapters:
- **`ClaudeCodeSubagentExecutor`** — detects host (env var / parent process / MCP context); uses Claude Code's Task tool (or `claude -p` in CI). Zero new API key required when running inside Claude Code.
- **`CodexExecExecutor`** — uses `codex exec` for non-interactive distillation. Same idea, different host.
- **`ByokLlmExecutor`** — direct API call (Anthropic / OpenAI / etc.) using user-configured key in `hub.yaml`. The fallback for headless / CI / "I don't want to run inside an agent."

Distillation contract (wire format) is in §7.

### 5.4 Family: storage adapters

> **SHIPPED** in 3.0.0 (Phase 1).

Two adapters split from the current `file_know_store`:
- **`FileCanonicalStore`** — read/write `canonical/<concept>/`. Handles snapshot directories.
- **`FileArtifactStore`** — read/write `artifacts/<kind>/<pack>/`. Handles incremental file-hash updates.

### 5.5 Hub resolver (extended)

> **SHIPPED** in 3.0.0 — project + user resolution active; package and remote layers stubbed.

```dart
abstract interface class HubResolver {
  Future<String?> resolveCanonical(String conceptId);    // walks project → user → (package) → (remote)
  Future<String?> resolveArtifact(String packName);      // project hub only

```

### 5.6 Services (split from `default_ae_know_service`)

> **SHIPPED** in 3.0.0 (Phase 4A). All five services landed; `default_ae_know_service` was deleted in the 3.0.1 hard cut.

- **`CanonicalService`** — scaffold, snapshot, distill (delegates to executor), upsert, list, diff, import-from-package.
- **`ArtifactService`** — ingest (calls heuristic extractor), sync (incremental re-scan), verify (per-pack), link (set references_canonical), upgrade-canonical, materialize (auto-add matrix rows from referenced canonical features).
- **`DriftService`** — compute code↔artifact drift (file hash diff vs `meta.yaml`); compute artifact↔canonical drift (which canonical invariants lack tests).
- **`DistillationService`** — orchestrates: build task → dispatch executor → validate output against schema → merge into canonical.
- **`HubService`** — extended for new resolution chain.

## 6. Behavior — major commands

### 6.1 `ae init`

> **SHIPPED** in 3.0.0.

In a directory: detect manifest files (pubspec.yaml / Cargo.toml / Package.swift / build.gradle.kts), recursively walk to find sub-packages, dispatch each to the matching `HeuristicExtractor`, write artifacts to project hub. No LLM. Sub-second per package; total time linear in package count.

### 6.2 `ae sync`

> **SHIPPED** in 3.0.0 (re-scan + drift); `--prune` was deferred and **SHIPPED in 3.1.0** (returns `pruned: [...]` in the envelope alongside drift; available via CLI and MCP `ae_sync` `prune` parameter).

Re-scan source files for each artifact; update file hashes in `meta.yaml`; write `drift.yaml`. Optional `--prune` removes artifacts whose source no longer exists. No LLM.

### 6.3 `ae status` (project-wide cockpit)

> **SHIPPED** in 3.0.0 (tier-classified output). Polished cockpit UI (color, paging) and priority weighting remain on the 3.1+ roadmap (see §15).

Walk all artifacts. Walk the `requires:` graph. Compute downstream-demand counts. Output **tier-classified gap report** (filtered by default; `--show all` for firehose):

- **Tier 1 — INVARIANT VIOLATIONS:** canonical asserts invariant; no test verifies it.
- **Tier 2 — UPSTREAM BLOCKERS:** features missing/partial in artifacts that downstream depends on. Sorted by downstream-count.
- **Tier 3 — PARTIAL FEATURES:** referenced canonical features at status `partial`.
- **Tier 4 — UNREFERENCED CANONICALS:** available but not linked. Includes "new since last verify" for newly-arrived canonicals.

`ae status --pack <name>` narrows to one artifact.

### 6.4 `ae artifact verify --pack <name>`

> **SHIPPED** in 3.0.0.

Single-artifact tiered verify (same tier scheme). Used in CI with `--strict` (exits non-zero on Tier 1+2 unless explicitly accepted via `drift.yaml`).

### 6.5 `ae artifact link --pack X --canonical Y[@v1]`

> **SHIPPED** in 3.0.0.

Adds canonical to artifact's `references_canonical`. Optionally locks to a snapshot. After link, `ae sync` materializes matrix rows for the new canonical features (default `impl: missing`).

### 6.6 `ae artifact upgrade-canonical --pack X --canonical Y --to v2`

> **SHIPPED** in 3.0.0.

Show diff between current locked version and target. Apply: rename rows, add new rows, remove obsolete, surface invariant changes. User reviews `drift.yaml` for the result.

### 6.7 `ae canonical scaffold --from-artifact <pack> [--from-artifact <pack2> ...] --concept <slug>`

> **SHIPPED in 3.1.0** (was a documented gap in 3.0; Iter 0 dogfood recommendation 4). CLI and MCP (`ae_canonical` op `scaffold`) both wired. Parses each artifact's `## Public API` section and emits one stub feature per detected symbol with `spec`/`invariant` placeholders; supports `--title <t>` and `--overwrite`. Multi-artifact composition (the `--from-artifact a,b,c` form combined with multi-pack distillation) remains on the roadmap (§15).

Heuristic, no LLM: extract candidate feature IDs from public symbols and doc-comment titles in one or more artifacts; emit a draft canonical pack with stub `spec` and `invariant` rows. User edits to make it real. **The make-or-break solo-dev command.**

### 6.8 `ae canonical distill --pack <artifact> --concept <slug>`

> **SHIPPED** in Phase 4E (3.0.0). CLI/MCP surface emit `executor_used` and `feature_count` in the envelope.
> **CHANGED in 3.0.2:** `mergeDistillation` surfaces duplicate-id collisions and emits both `feature_count_received` and `feature_count_after_merge` with a warnings list when the two diverge (Iter 0 dogfood bug 4).

Dispatch distillation: build a `DistillationTask` from the artifact, hand to the matched executor, receive validated `DistillationOutput`, merge into canonical. Either creates a new canonical or refines an existing one (via `--mode upsert | refine`).

### 6.9 `ae canonical snapshot --concept X`

> **SHIPPED** in 3.0.0.

Freeze current `canonical/<concept>/` into `canonical/<concept>/v<n>/`; bump `meta.yaml.version`. Run only when introducing a breaking change. Optional flag `--migration-doc` opens an editor to write `migration_to_v<n+1>.md`.

### 6.10 `ae canonical import --from <path> [--to <hub>]`

> **SHIPPED** in 3.0.0. Auto package-hub discovery remains roadmapped (§14, §15).

Manual entry point for package-shipped canonicals (until 3.x auto-discovery). Copies a canonical from `<package>/.ae_hub/canonical/<concept>/` into the target hub (project by default).

## 7. Distillation contract (wire format)

> **SHIPPED** in 3.0.0 (Phase 3). Wire format below is unchanged.
> **CHANGED in 3.1.0:** `DistillationService.distill` now returns `DistillationResult { output, executorId }` instead of bare `DistillationOutput`. CLI/MCP envelopes report `executor_used` directly from the result rather than re-running `canRun()` on every executor (Phase 4E quirk). The wire format between AE and the executor is unchanged.

Same shape regardless of executor.

```jsonc
// AE → executor (input)
{
  "task": "distill_pack",
  "schema_in": "ae.distillation.task.v1",
  "schema_out": "ae.canonical.draft.v1",
  "concept_id": "ecsly/render_pipeline",
  "concept_version": 1,
  "source_artifact": {
    "name": "dart_ecs_render3d_core",
    "language": "dart",
    "files": ["lib/src/passes/basic.dart", "lib/src/scene.dart", ...],
    "structural_summary": "<from existing index.md>"
  },
  "matrix_seed_rows": [/* optional canonical rows already present */],
  "examples": [/* few-shot from prior accepted distillations */]
}

// executor → AE (output, validated against ae.canonical.draft.v1)
{
  "schema": "ae.canonical.draft.v1",
  "concept_id": "ecsly/render_pipeline",
  "concept_version": 1,
  "index_md": "...condensed concept doc, ~600 words...",
  "matrix": {
    "column_schema": [{ "id": "spec", "type": "text" }, ...],
    "features": [
      { "id": "render.scene_extract", "spec": "...", "invariant": "..." },
      ...
    ]
  },
  "patterns_md": "...optional impl-specific idioms..."
}
```

On schema validation failure: AE retries once with the validation error included as additional context. Second failure → fail with clear diagnostic; no silent partial merge.

## 8. Drift detection — both axes

> **SHIPPED** in 3.0.0 (Phase 4C). Hand-edited `accepted:` block ships; the proposed `ae artifact accept-drift` interactive command remains on the 3.1+ roadmap (§15).

- **Code drift** (artifact ↔ source files): `ae sync` compares current file hashes against `meta.yaml.files`. Output: added / modified / removed files. Pack flagged "scan stale" until `ae sync` runs.
- **Intent drift** (artifact ↔ canonical): `ae artifact verify` walks each canonical referenced by the artifact; for each canonical feature with a declared invariant, check the artifact has a test asserting it. Output: invariants without tests = Tier 1.
- **Accepted drift**: `drift.yaml` has an `accepted:` section users can hand-edit to mark known-and-OK drift. `verify --strict` excludes accepted drift from failure.

## 14. Anti-goals — 3.0 deliberately does NOT ship

> **STILL ACCURATE** as of v3.1.0. None of the listed anti-goals were crossed. (`ae canonical distill` is delegation, not codegen — it produces canonical packs, not target-language source.)

- **Cross-language code generation.** LLMs aren't reliable for idiomatic ports of non-trivial code in 2026. The matrix-as-language-neutral-spec is the seed; full generation is post-3.x at earliest.
- **Auto-sync between code and canonical.** Drift VISIBILITY is honest; auto-sync would lie. Stays 3.0; auto-sync stays out.
- **Public canonical hub.** Pack format is publishable (no machine-specific paths, deterministic IDs, attribution required), but no remote sharing system. Roadmapped.
- **Multi-repo workspaces.** One `.ae_hub` per repo; user hub for shared canonical. No cross-repo aggregation in 3.0.
- **Team / sharing flows.** Solo-dev focus.
- **Agent teams** (Claude Code's agent-teams docs the user shared) — reserved for post-3.x richer agent integration.
- **Dgraph or any graph backend.** Filesystem packs already model a graph via stable IDs and references. Dgraph wire-up is a post-3.x knob.
- **Auto package-hub discovery.** Resolver supports the layer; manual `ae canonical import` is the 3.0 path.
- **Migration command** for old `know/` layouts. Hard cut; old hubs error out with clear instructions.

### Resolved by Iter 1 dogfood ([report](../notes/2026-04-27-ae-3.0-dogfood-iter-1.md))

- ~~Re-run distill determinism check~~ — **measured: drifting badly.** Two consecutive runs on the same artifact produced 40 vs 28 features with **zero overlapping ids**. Resolution path: [id-stability design](./2026-04-27-canonical-id-stability-design.md) — distill never invents ids; scaffold or accept-concept does.
- ~~Larger-pack distillation scaling~~ — **measured: output is roughly constant-cost.** 82 files → 27 features in less wall time than 8 files → 28 features. Distill caps its output regardless of input. Resolution path: same id-stability design — scaffold-then-distill becomes the documented workflow for non-trivial packs; distill enriches the symbol-derived skeleton instead of re-deriving from source.
- ~~Distill behavior without `claude` on PATH~~ — **measured: dispatcher commits to `claude_code`, no `codex` fallback.** Failed in 17ms with `executor claude_code failed twice`. CI hazard. Tracked separately from id-stability; see "Open from Iter 1" below.
- ~~Compare distilled canonicals against hand-written references~~ — **measured: zero contradictions, three docs gaps.** [`cli-reference.md`](../../../docs_site/docs/ae-3/cli-reference.md) is missing `### ae generate`, `### ae verify`, `### ae evaluate` sections. Tracked separately as a docs task.

### Carry-forward from Iter 1 dogfood (open, partly addressed by id-stability design)

- **Distillation dispatcher needs executor fallback.** When `claude` is missing from PATH but `codex` is registered, dispatcher should fall through; today it commits to `claude_code` and fails. Spec §6.5 (executor selection) needs a defined fallback chain. Independent of id-stability work.
- **`ae doctor` should probe every registered executor binary.** Today it checks `codex_available` but not `claude_available`. Asymmetric. Independent of id-stability work.
- **Pubspec versions are stale relative to CHANGELOG.** Iter 1 binary reported `cli=3.0.0 core=3.0.0` despite 3.0.1 / 3.0.2 / 3.1.0 entries in [CHANGELOG.md](../../../CHANGELOG.md). Bump on next release.
- **Periodic distill-vs-cli-reference comparison could be CI** (speculative — Q5 found three missing sections in <30 min of inspection; an automated coverage diff would catch the next gap immediately).

### Resolved-direction proposed by Iter 1 (see [id-stability design](./2026-04-27-canonical-id-stability-design.md))

The Q1/Q2/Q3 findings together imply that the canonical workflow needs three structural changes:

1. **Distill never invents ids.** Hard validator + one-sentence prompt constraint. Closes Q1.
2. **`ae canonical scaffold --update`** for source-symbol reconciliation; `ae canonical accept-concept` for cross-cutting invariants. Replaces "distill writes anything it likes" with proposal-then-accept for non-symbol features.
3. **`canonical/<concept>/DESIGN_FAQ.md` as executor context.** Pins cross-run vocabulary. Format borrowed from ecsly's plugins-DESIGN_FAQ pattern.

Reuse classification (`library` vs `app`) is treated as a derived view over the artifact-link graph, not as a row-level field. Granularity is a side effect of the symbol-vs-concept split, not a separate decision. See [id-stability design](./2026-04-27-canonical-id-stability-design.md) Q&A for full rationale.

Implementation plan: [`plans/2026-04-27-canonical-id-stability-impl.md`](../plans/2026-04-27-canonical-id-stability-impl.md) (3 phases, each shippable on its own).

## 16. Open questions (to discuss in spec review)

1. **Canonical concept-id namespace conventions.** Project-private uses `<project>/<concept>` (e.g., `ecsly/render_pipeline`). Public uses bare `<concept>` (e.g., `ecs`). Should there be a strict scheme (`public/<concept>` vs `private/<project>/<concept>`)? Or by convention only?
2. **`drift.yaml.accepted` UX.** Hand-editing YAML is friction. Should we have `ae artifact accept-drift --feature <id> --reason <text>`?
3. **`ae status` priority weighting.** "Sorted by downstream-count" is a start. Should the user be able to attach manual priority tags to features (e.g., `priority: critical`)?
4. **Distillation cost preview.** `ae canonical distill --estimate` for direct-LLM executor is straightforward (token count). For subagent executors (Claude Code Task), cost is the user's subscription — should we still try to estimate, or just say "this dispatches N subagent tasks"?
5. **`canonical.json` for machine consumption.** Currently canonical is YAML/MD only. Should we also emit a normalized JSON (the way spec_export does today) for non-YAML consumers? Probably yes, automatically.
6. **Artifact lifecycle status** (`active | archived | abandoned`) for prototypes that come and go (raised in brainstorm; not folded into 3.0). 3.0 default: prototypes are deleted from `.ae_hub/artifacts/local/` by hand when retired. Defer formal lifecycle status to 3.1 — confirm or pull forward?
