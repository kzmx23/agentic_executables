## 2. Q1 — Determinism

Two consecutive distill runs on `agentic_executables_cli`, same binary,
same artifact, no shared state.

| metric | Run A | Run B |
|---|---|---|
| feature_count | 40 | 28 |
| wall_seconds | 277 | 256 |
| envelope == matrix | yes | yes |
| warnings | none | none |

Set diff of feature ids: **0 common, 40 only-in-A, 28 only-in-B.**

The variance is two-layered:
1. **Granularity drift.** Run A documented every subcommand as its own
   feature (40); Run B grouped by command family + cross-cutting layers
   (28).
2. **Namespacing drift.** Run A used `ae_cli.<symbol>` (matching spec
   §6.7's heuristic-scaffold convention). Run B used semantic
   categories — `output.* command.* engine.* io.* resources.*` —
   ignoring the spec convention.

Concept overlap is high (both runs cover JSON envelope, human mode,
init/status/sync/doctor/canonical/artifact/spec) but the id sets
disjoin completely.

**Verdict: drifting.** Not noise — the executor picks a different
*partition* of the surface each run.

## 3. Q2 — Scaling on `agentic_executables_core`

Target: 82 Dart files (~10× the input of `agentic_executables_cli`).

| pack | files | features | wall (s) | features/file |
|---|---|---|---|---|
| `agentic_executables_mcp` | 3 | 21 | 187 | 7.0 |
| `agentic_executables_cli` (Run B) | 8 | 28 | 256 | 3.5 |
| `agentic_executables_core` | 82 | 27 | 199 | 0.33 |

Ratios (core / cli): input 10.25×, features 0.96×, wall 0.78×.

**Distill output is roughly constant-cost regardless of input size.**
A 10× increase in input files produced no increase in feature count
and even a slight DECREASE in wall time. The executor is not reading
proportionally — it is sampling and/or operating on a fixed-size
output budget.

For `agentic_executables_core` specifically, the 27 features land at
the **subsystem level** (`architecture.* artifact.* canonical.*
distillation.* doctor.* drift.* generation.* heuristic.* hub.*
package.* result.* verify.*`) — one feature per major abstraction.
Sensible as a top-down summary, but **categorically different** from
how distill treated the smaller packages: mcp got per-tool enumeration
(7 features/file), cli got per-command (3.5 features/file), core got
per-subsystem (0.33 features/file).

The same `distill` command produces three structurally different
kinds of canonical depending on input size. No spec contract pins
which kind to expect.

**Verdict: sub-linear** — but for the wrong reason. The executor
doesn't ceil out from too much input; it caps its output regardless
of input. For large packages, this means lossy summarization rather
than a graceful failure or warning.

Hypothesis for Iter 2: running `ae canonical scaffold --from-artifact
agentic_executables_core` (3.1.0 feature) before distill would seed
the matrix with per-symbol features, and Q3's preservation behavior
should keep them when distill enriches. Worth testing.

## 4. Q3 — Snapshot + upsert preservation

Starting state: Q1 Run B's matrix (28 features). Procedure:
`ae canonical snapshot` then `ae canonical distill --mode upsert`.

| metric | pre | post |
|---|---|---|
| feature_count | 28 | 28 |
| wall_seconds | (256 baseline) | 151 |

Set comparison: 28 retained, 0 added, **0 dropped**.

Two findings:
1. `ae canonical snapshot` **moves** the live matrix into `vN/`; it
   does not copy. Post-snapshot there is no live `matrix.yaml` at the
   concept root until the next distill writes one.
2. The 151s wall (vs 256s baseline) suggests the executor read the
   `v1/` snapshot directory as context and re-emitted its structure,
   rather than re-deriving from the artifact. So preservation works
   today, but **for the wrong reason** — through executor convenience,
   not merge-side enforcement.

**Verdict: upsert preserves.** With the caveat that the mechanism is
fragile.

## 5. Q4 — Dispatcher with `claude` hidden

Hid `claude` surgically by renaming the symlink (no PATH munging).
Codex remained available. Re-ran `ae canonical distill`.

```json
{"success":false,"command":"canonical distill","data":{},
 "error":{"code":"distillation_failed",
          "message":"executor claude_code failed twice: failed to invoke claude; failed to invoke claude"},
 "warnings":[],"meta":{"timing_ms":17,"versions":{"cli":"3.0.0","core":"3.0.0"}}}
```

**Outcome (c) per the plan: dispatcher tried `claude_code` anyway,
retried twice, did not fall back to `codex`.** Failed in 17ms.

Also notable: `ae doctor` has no `claude_available` check (it has
`codex_available`, `dart_available`, `skill_target_writable`,
`registry_reachable`). The operator cannot validate the distillation
toolchain before running distill.

**Verdict: fails confusingly.** A CI runner with only `codex` will be
unable to distill, despite the dispatcher having a working fallback
target available.

## 8. Roll-up — items for spec §15

1. **Distillation prompt must constrain id namespacing and granularity.**
   Q1 showed the executor invents both. Either pin it in the prompt or
   pre-seed the canonical via `ae canonical scaffold` before distill so
   the executor has examples to copy.
2. **`canonical snapshot` should copy, not move.** Today it moves the
   live matrix into `vN/` and the next distill regenerates the live
   slot. Document the move semantics in
   [cli-reference.md `### ae canonical snapshot`](../../../docs_site/docs/ae-3/cli-reference.md), or change the
   command to copy-and-bump. (Q3 finding 1.)
3. **Upsert preservation should be merge-side, not executor-side.**
   Q3 preserves features today only because the executor reads the
   `vN/` snapshot dir as context. Add a test that re-distills against
   a known matrix and asserts no id is dropped. (Q3 finding 2.)
4. **Distillation dispatcher needs executor fallback.** Q4 showed
   dispatcher commits to `claude_code` and fails when `claude` is
   missing, even with `codex` available. Spec §6.5 (executor selection)
   needs a defined fallback chain.
5. **`ae doctor` should probe every registered executor binary.**
   Q4 showed `codex_available` exists but not `claude_available`.
   Symmetrize. (Q4 finding 2.)
6. **`cli-reference.md` is missing `ae generate`, `ae verify`,
   `ae evaluate` sections.** Q5 finding. Fix is straightforward — the
   canonical's specs for those commands are draft-quality.
7. **Pubspec versions are stale relative to CHANGELOG.** Iter 1's
   binary reports `cli=3.0.0 core=3.0.0` despite 3.0.1 / 3.0.2 / 3.1.0
   release entries in CHANGELOG. Bump on next release.
8. **Periodic distill-vs-cli-reference comparison could be CI.**
   Q5 found 3 missing sections in <30 min of inspection. An automated
   coverage diff would catch the next gap immediately. (Speculative,
   not a hard ask.)
9. **Distill output budget needs a spec contract.** Q2 showed feature
   count is roughly fixed (21-28) regardless of input size (3-82
   files). For large packages this is lossy summarization. Either
   document the actual budget or scale features with input.
10. **For large packages, scaffold-then-distill should be the
    documented workflow.** Q2's flat output budget means
    `canonical init → distill` will drop most of the public surface
    on a large pack. The `ae canonical scaffold --from-artifact`
    workflow (3.1.0) plus Q3's preservation behavior should preserve
    the scaffolded entries while distill enriches. **Iter 2 should
    test this hypothesis directly.**

## 10. Postscript — design response

The §8 roll-up (Q1, Q2, Q3, Q9, Q10) turned out to share a common root cause: distill is permitted to invent feature ids freely, and the canonical workflow has no contract that pins them across runs. After discussion, the resolution direction is captured in
[`specs/2026-04-27-canonical-id-stability-design.md`](../specs/2026-04-27-canonical-id-stability-design.md), which proposes:

1. **Distill never invents ids** — hard validator on the merge step, one-sentence prompt constraint.
2. **Three id-discovery channels** — source-symbol (deterministic, via `scaffold --update`); concept-proposal (LLM proposes, human accepts via `accept-concept`); hand-authored (escape hatch).
3. **DESIGN_FAQ.md per concept** as executor context for cross-run vocabulary pinning.
4. **Reuse classification as a derived graph property**, not a row-level field.

Implementation is sequenced as three shippable phases in
[`plans/2026-04-27-canonical-id-stability-impl.md`](../plans/2026-04-27-canonical-id-stability-impl.md).

**Update 2026-04-28:** Phase A and Phase B are **shipped**. Q1's drift root cause is empirically closed: two consecutive scaffold-then-distill runs against `agentic_executables_cli` produce 14/14 identical ids in both Phase A ([smoke](../notes/phase-a-smoke/SUMMARY.md)) and Phase B ([smoke](../notes/phase-b-smoke/SUMMARY.md)). Phase C (FAQ-as-context, reuse-class derivation, migration helper) and Iter 2 dogfood remain proposed.

Iter 1 §5 (Q4 — dispatcher fallback) and §6 (Q5 — three missing reference sections) are tracked separately in spec [§15 carry-forward](../specs/2026-04-17-ae-3.0-design.md) — they don't share the id-stability root cause.
