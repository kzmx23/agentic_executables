**Date:** 2026-04-27
**Repo SHA (v2):** `a4db8f2070e055e520a15fa86fdb7fd241af9b58`
**Branch under test:** `v2`
**Side branch (canonical artifacts):** `dogfood-iter-0` @ `0af0eeb7ca5846cdf8c0968110dbfec4d1935499`
**Operator:** Claude Code subagent, single-pass observation run.

The premise: re-bootstrap an AE 3.0 hub on the AE CLI repo itself, run
`ae init` followed by `ae canonical distill --pack agentic_executables_cli
--concept ae_cli`, and measure what actually happens. Spec §14 forbids
cross-language code generation, so we are explicitly probing whether the
canonical produced is a usable spec-of-the-CLI vs. just a structural echo.

## 2. Iter 0 procedure

In order, with logs stashed in
`docs/superpowers/notes/dogfood-iter-0/` on the side branch:

1. `mv .ae_hub .ae_hub.v2-preserve` (preserve the v2 hub).
2. `dart compile exe …/ae.dart -o /tmp/ae-3.0-iter0`.
3. `ae hub init --path .` → creates `know/`, `packages/`, `use/`,
   `hub.yaml` **directly at `--path`, not nested under `.ae_hub/`**
   (see §5 below — first surprise).
4. Cleaned up; `ae hub init --project` → properly creates `.ae_hub/` at
   the project root.
5. `ae init --root .` → ingests three Dart packages.
6. `ae canonical init --concept ae_cli --title "AE CLI" --root .` →
   scaffolds 3 files.
7. `ae canonical distill --pack agentic_executables_cli --concept ae_cli
   --root .` → success.
8. Snapshot to `docs/superpowers/notes/dogfood-iter-0/`, force-add
   canonical (`.ae_hub/` is gitignored), commit on `dogfood-iter-0`.
9. Restore the original v2 hub on `v2`. Tree clean modulo `.DS_Store`.

## 4. What worked

- `dart compile exe` of the CLI: 2.4s, zero warnings, single static binary.
- `ae hub init --project` produces a valid v3 hub layout (`.ae_hub/{know,packages,use}/`, `hub.yaml`).
- `ae init --root .` scans, identifies 3 Dart packages by pubspec, ingests in 49ms — extremely cheap and idempotent.
- `ae canonical init` works as the documented scaffold fallback.
- **`ae canonical distill` ran end-to-end via the `claude_code` executor.** Nested Claude Code subagent invocation works inside an existing Claude Code session (CLAUDECODE=1). 4m39s for a 9-file Dart package is reasonable.
- Output canonical is genuinely useful: 58 features with real type signatures, exit-code semantics, retry/extractor strategy notes. Not a structural echo of the heuristic index.
- Dispatcher selected `claude_code` even though `codex` and (presumably) other executors were available — preference order is sane for this environment.

## 5. What didn't (or surprised)

1. **`ae hub init --path .` (no `--project`) creates a naked hub at the path argument**, not nested under `.ae_hub/`. So running it at the repo root pollutes the worktree with `know/`, `packages/`, `use/`, and `hub.yaml` directly. The error returned by `ae init` correctly directs the user to `--project`, but the success of the other invocation is misleading. (`agentic_executables_cli/lib/src/cli.dart` `hub init` handler — `--project` adds a `.ae_hub/` segment.)

2. **`ae canonical distill --help` returns "No contextual help found"**. The handler exists and runs, just no `--help` text. Discoverability gap.

3. **Top-level `ae --help` listing is incomplete**. `init`, `status`, `sync`, `canonical`, `artifact`, `spec` are all dispatchable but absent from the bare `--help` output. They show up via `ae <name> --help` (`canonical --help`, `init --help` work). Likely a CommandRunner subcommand-vs.-handler enumeration bug.

4. **`feature_count` envelope field disagrees with matrix.yaml row count** (56 vs. 58). Lightweight off-by-N somewhere in `DistillationService` envelope assembly.

5. **`canonical init` writes a `column_schema` of `[spec, invariant]`, but distill emits features with `invocation` and `notes` columns too**, not declared in the schema. Either the column_schema needs widening on init (recommend), or the distillation prompt needs tightening to match it. Today the matrix is technically out-of-spec against its own `column_schema`.

6. **`canonical scaffold --from-artifact` is documented in spec §6.7 but not wired** (also called out in the cli-reference's Surprises section). The init-then-distill flow is the working substitute today, which works, but means there is no auto-seeded matrix from the heuristic index — distill starts from scratch each run.

7. **`.ae_hub/` is gitignored, so committing the canonical to a side branch requires `git add -f`.** Not a bug per se, but worth documenting in the dogfood playbook — the `git add` step in the brief silently does nothing without `-f`.

## 6. Recommendations for AE 3.1

Ordered by impact, none contradicting spec §14 (no cross-language code-gen anti-goal):

1. **Fix `feature_count` discrepancy in `DistillationService`** — emit `len(matrix.features)` post-write rather than a counter that drifts. Trust property of the envelope is high-value; off-by-2 erodes it. (file: `agentic_executables_core/lib/src/canonical/distillation_service.dart`, search for `feature_count`).

2. **Make `ae hub init` consistent**: either default to `--project` semantics (always nest under `.ae_hub/`) or refuse to run without `--project` when `--path` resolves to a non-empty directory. Current behavior makes the user trash their cwd silently on the obvious first guess. (file: `agentic_executables_cli/lib/src/cli.dart`, `_handleHubInit`).

3. **Surface all dispatchable commands in `ae --help`**, including `init`, `status`, `sync`, `canonical`, `artifact`, `spec`. Today they exist as runnable but invisible — bad first-run UX.

4. **Wire `ae canonical scaffold --from-artifact <pack>`** as documented in spec §6.7, seeding the matrix with the heuristic-extracted symbol list. Distill then becomes an *enrichment* pass rather than a from-scratch one — should significantly cut input ambiguity and (probably) wall time on larger packs.

5. **Align scaffold's `column_schema` with what distill actually emits** (`spec, invocation, invariant, notes`), or alternatively gate distill output through the declared schema. Pick one direction; today they disagree.

(Deliberately *not* recommending: a hub.yaml `byok:` template stub. The brief allowed it, but BYOK absence didn't block this run — the `claude_code` executor sufficed. Adding BYOK templates without primary-source signal that users want it would be premature.)

## 7. Open questions for Iter 1+

- **Re-run distill on the same inputs.** Determinism check — does `claude_code` produce the same 58 features twice? If not, what's the variance? This matters for canonical stability across CI runs.
- **Distill `agentic_executables_core` (much larger pack).** Does wall time scale linearly, sub-linearly, or hit some pathological case? Does the executor session blow its context window?
- **`ae canonical snapshot` then second `distill --upsert`.** Spec says distill is upsert-shaped; verify that re-running on a snapshot doesn't drop existing features.
- **Without `claude` on PATH** (e.g. on CI without an installed Claude Code), what does dispatcher do? Today we can't tell from this run — both `claude` and `codex` were available. Iteration 1 should explicitly hide one and observe.
- **Compare canonical to `docs_site/docs/ae-3/cli-reference.md`** — does the distilled spec contradict the hand-written reference anywhere? Disagreement is a signal worth tracing.
- **Heuristic index quality on a Dart package without much exported API surface** (e.g. a thin façade package) — does distill have enough to chew on, or does it hallucinate?
